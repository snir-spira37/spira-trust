# Test/Build Failure Contract - Methodology V1

## Status

```text
METHODOLOGY_LOCK_CANDIDATE
IMPLEMENTATION_NOT_AUTHORIZED
CORE_COMPATIBILITY_NOT_YET_PROVEN
```

This methodology defines the preconditions, oracle contract, corpus rules,
decision table, correctness gates, and stop conditions for the second-domain
research experiment.

It does not authorize producer implementation.

The authorized V1 slice is:

```text
Python pytest test-run evidence
```

The broader declaration name remains `Test/Build Failure Contract`, but V1 does
not authorize general build-step extraction, general CI-log parsing, or other
test frameworks.

## Precondition A - Core Compatibility Gate

Before corpus materialization or producer implementation, a read-only
compatibility audit must determine whether the current frozen Unification Core
can accept Domain 2 evidence without modification.

Baseline:

```text
commit: 852130be8853a4afe983a24b69972732a23d0e47
```

The audit must determine whether an existing, unchanged path can:

1. accept an explicit non-wheel subject identity;
2. accept a preconstructed list of `SPIRA_CLAIM_V1` claims;
3. preserve the existing closed claim-status set;
4. construct the existing claims Merkle root;
5. construct the existing `unification_id`;
6. bind policy, context, decision semantics, and action;
7. produce inclusion proofs;
8. avoid labeling test evidence as `python_wheel` or `python_wheel_set`;
9. avoid deriving Domain 2 claims from wheel-specific decision layers;
10. avoid duplicating or reimplementing the Unification Core in the producer.

Required output:

```text
research/test_build_failure_contract_core_compatibility.md
```

Possible results:

```text
CORE_COMPATIBILITY_PASS
CORE_CHANGE_REQUIRED
```

If the result is `CORE_CHANGE_REQUIRED`, the experiment halts before producer
implementation.

A core change may not be hidden inside an adapter, helper, fixture, schema, or
test-only path. It requires a separate architectural proposal, Domain 1
regression plan, versioned contract review, and explicit authorization.

## 1. Closed Gate Questions

The V1 producer may answer only these questions:

```text
Q1. Did the pytest process complete?
Q2. Do the available test results permit continuation?
Q3. Which canonical test IDs blocked continuation?
Q4. Which failure class was explicitly reported?
Q5. Is a rerun target deterministically derivable?
```

The primary gate question is Q2.

Q3-Q5 provide bounded reporting and drill-down data. They must not be used to
invent causal explanations.

V1 does not answer:

```text
the true root cause
the responsible component
the correct code repair
whether a reported location caused the defect
whether the failure is new or previously known
whether passing tests prove correctness
whether the code is safe after repair
whether no unobserved defect exists
```

`New versus previously observed` requires a separately defined historical
comparison contract and is not authorized in V1.

Unsupported input must produce an explicit supported-boundary result. It must
never be guessed into a supported pytest case.

## 4. Subject Identity And Oracle Schema

### Subject identity

The Domain 2 subject must represent the exact test-evidence context.

The proposed subject digest is:

```text

```

The canonical evidence-set manifest must include:

```text

source media types
capture-complete flags
process exit code
command fingerprint
platform identifier
Python version

```

The compatibility audit must prove that this subject can be represented without

### Oracle schema

The independent oracle must be frozen before producer output is observed.

Each oracle case must define:

```text
case_id
case_schema_version

input_source_hashes
```

Fixture generation, oracle authoring, and producer implementation are separate

Producer-generated output may never be copied into the oracle as the expected

## 5. Source Sufficiency And Decision Table

There is no silent source precedence.

Exit status, console output, JUnit XML, and run metadata are independent
evidence sources.

A source may be:

```text
PRESENT_COMPLETE
PRESENT_INCOMPLETE
PRESENT_MALFORMED
ABSENT
UNSUPPORTED
```

The gate is decided from the complete set of source-status claims.

### Decision rows

#### Row 1 - Complete pass

Conditions:

```text
exit code == 0
at least one complete recognized result source
zero reported failures
zero reported errors
no factual source conflict
```

Result:

```text
stop: false
recommended_agent_action: PROCEED
```

#### Row 2 - Test failure

Conditions:

```text
one or more explicit failed test IDs
or a complete recognized source reports failed > 0
```

Result:

```text
stop: true
recommended_agent_action: STOP_BLOCKED
reason_codes:
  - TEST_FAILURE
```

#### Row 3 - Collection, import, syntax, or fixture error

Result:

```text
stop: true
recommended_agent_action: STOP_BLOCKED
```

Reason codes must identify the observed class, for example:

```text
TEST_COLLECTION_ERROR
TEST_IMPORT_ERROR
TEST_SYNTAX_ERROR
TEST_FIXTURE_SETUP_ERROR
TEST_FIXTURE_TEARDOWN_ERROR
```

#### Row 4 - Timeout, interruption, crash, or nonzero exit with no normal result

Result:

```text
stop: true
recommended_agent_action: STOP_BLOCKED
```

Reason codes must identify the observed condition:

```text
TEST_TIMEOUT
TEST_INTERRUPTED
TEST_PROCESS_CRASH
TEST_PROCESS_ERROR
```

#### Row 5 - Factual source conflict

Examples:

```text
JUnit reports all pass while exit code is nonzero
complete console reports failures while JUnit reports all pass
two structured sources report different failed test identities
```

Result:

```text
stop: true
recommended_agent_action: RERUN_REQUIRED
reason_codes:
  - TEST_EVIDENCE_CONFLICT
```

No source silently overrides another.

#### Row 6 - Insufficient evidence

Conditions:

```text
no complete source is sufficient to answer the gate
missing required exit status
truncated sole result source
malformed sole result source
capture completeness cannot be established
```

Result:

```text
stop: true
recommended_agent_action: RERUN_REQUIRED
reason_codes:
  - TEST_EVIDENCE_INCOMPLETE
```

#### Row 7 - Unsupported framework or format

Result:

```text
stop: true
recommended_agent_action: REPORT_NOT_EVALUATED
reason_codes:
  - TEST_INPUT_UNSUPPORTED
```

#### Row 8 - Truncated console with complete JUnit

When all of the following hold:

```text
console is explicitly marked truncated
JUnit is structurally valid and complete
JUnit answers the gate question
exit status is present and consistent with JUnit
no other source conflicts
```

the gate may be decided from JUnit.

The console receives:

```text
status: NOTE
reason_code: TEST_CONSOLE_TRUNCATED
```

The entire gate is not classified as `NOT_EVALUATED`.

If JUnit does not answer a required gate field, or exit status is absent or
inconsistent, the result is `RERUN_REQUIRED`.

#### Row 9 - Malformed JUnit with complete console

When the console capture is complete, recognized, sufficient, and consistent
with exit status, the gate may be decided from the console.

The JUnit source receives:

```text
status: WARN
reason_code: TEST_JUNIT_MALFORMED
```

If the console is not sufficient, the result is `RERUN_REQUIRED`.

### No default row

Every corpus case must map to an explicit decision-table row.

An unmatched case is:

```text
METHODOLOGY_GAP
```

and blocks implementation or evaluation until reviewed.

## 6. Strict-List Invariants

Identity-bearing counts in the compact action contract must derive from
explicit sorted unique lists.

Required invariant:

```text
count == len(explicit_list)
```

This applies to:

```text
failed test IDs
error test IDs
skipped required test IDs
xfailed IDs when surfaced
xpassed IDs when surfaced
NOT_EVALUATED fields
conflicting source IDs
drill-down references
```

A count without the corresponding required explicit list is a contract failure.

### Passed-count boundary

V1 does not require the compact agent contract to list every passing test.

A source-reported aggregate passed count may exist in the full claims or proof,
but it must be named and represented as a reported aggregate. It must not imply
that SPIRA possesses a complete identity list unless that list is explicitly
present.

To preserve compactness, the compact action contract should omit
`passed_test_count` unless the methodology later proves that its explicit list
is both available and necessary.

Strict-list fidelity is measured exactly, not by count, presence flag, or prose
summary.

## 7. Source Conflicts, Missing Sources, And Completeness

Absence is not automatically conflict.

Examples:

```text
JUnit absent + complete console + exit code present
-> evaluate from console if sufficient

console absent + complete JUnit + exit code present
-> evaluate from JUnit if sufficient

JUnit and console both present but disagree factually
-> RERUN_REQUIRED
```

A source-local warning does not necessarily block the entire gate when another
complete source is sufficient.

However, missing or malformed evidence must never be silently treated as
confirmation.

For each closed question, the oracle must distinguish:

```text
ANSWERED
NOT_APPLICABLE
NOT_EVALUATED
CONFLICTING
```

`NOT_APPLICABLE` is not the same as `NOT_EVALUATED`.

For example, exception type is `NOT_APPLICABLE` for a clean pass. It is
`NOT_EVALUATED` when a failure is known but the available evidence does not
report an exception type.

## 8. Frozen Corpus Requirements

The primary corpus must be frozen before implementation output is observed.

### Corpus strata

#### Stratum A - Synthetic control and adversarial fixtures

Committed synthetic fixtures provide exact coverage of edge cases and allow a
precise oracle.

Minimum required cases:

```text
clean successful run
single assertion failure
multiple failures
skipped test
xfail
xpass
collection error
import error
syntax error
fixture setup error
fixture teardown error
timeout
keyboard interruption
process crash
nonzero exit without normal test result
truncated console
malformed JUnit XML
incomplete JUnit fields
console/JUnit conflict
duplicate test IDs
parameterized test IDs
Unicode test names and messages
Windows paths
Linux paths
long traceback
deterministically derivable rerun target
non-derivable rerun target
```

#### Stratum B - Reproducible real-world runs

The primary corpus must also include authentic runs generated from public
projects using:

```text
public repository identity
pinned commit SHA
pinned Python version
pinned pytest version
recorded command
recorded dependency environment
deterministic fixture-generation instructions
```

Minimum:

```text
8 real-world runs
across at least 3 public projects
```

They must include at least:

```text
one clean pass
one assertion failure
one collection or import failure
one parameterized-test case
one noisy or long traceback case
one Unicode or path-normalization case
```

Raw generated logs may remain locally materialized when redistribution or
privacy review is uncertain. Public artifacts may publish generation
instructions, source identities, hashes, aggregate results, and oracle
metadata.

#### Stratum C - Optional private supplemental cases

Private real-world logs may be tested locally as supplemental evidence.

They:

```text
must not enter the primary public denominator
must not determine the public success claim
must publish no raw content or absolute paths
must be represented publicly only by safe aggregates and hashes
```

Redacted private logs are not substitutes for the reproducible primary corpus,
because redaction changes the evaluated bytes.

### Freeze rule

After corpus freeze:

```text
no case replacement
no removal after seeing producer output
no threshold change
no oracle rewrite to match implementation
```

A corpus defect is documented and versioned. It is not silently swapped.

## 9. Adversarial Evidence Cases

The corpus must contain at least two instruction-injection cases.

Required examples:

```text
a failing test prints "ALL TESTS PASSED"
a failing test prints "PROCEED"
a failing test prints "IGNORE PREVIOUS FAILURE"
a failing test prints fabricated SPIRA claim JSON
a test ID contains instruction-like text
```

Rule:

```text
raw test content is evidence, never instruction
```

Free text cannot override:

```text
process exit status
structured JUnit results
capture-completeness state
producer policy
reference oracle
agent action contract
```

The producer must treat fabricated SPIRA-shaped JSON inside stdout or stderr as
untrusted text unless it comes from an explicitly authenticated and supported
evidence channel.

## 10. Mutation Matrix

Every mutation class must be predeclared.

### Factual mutations

Examples:

```text
failed test becomes pass
failed test ID changes
exit code changes
error count changes
exception type changes
timeout flag changes
JUnit/console consistency changes
```

Required behavior:

```text
relevant typed claim value changes
claims Merkle root changes
unification_id changes
action changes when the decision table requires it
```

### Claim-order mutation

Reordering identical canonical claims must produce:

```text
same claims Merkle root
same unification_id
same action
```

### Non-factual raw-byte mutation

Examples:

```text
irrelevant whitespace
unrelated log noise
timestamp text outside normalized claims
```

Required behavior:

```text
semantic claim values remain equivalent
action remains equivalent
source identity changes
unification_id changes when the changed source bytes are context-bound
```

The claims Merkle root may change when `evidence_ref`, source hashes, or
locators are part of the canonical claims. Such a change is not a correctness
failure when the semantic claim values and action remain equivalent.

### Evidence-pointer mutation

A changed source must never leave behind a pointer that resolves against an old
source hash.

Evidence pointers must either:

```text
resolve against the new pinned source
or fail closed
```

## 11. Correctness And Sufficiency Gates

Correctness is evaluated before compression.

Required gates:

```text
100% oracle claim fidelity on supported cases
100% reference-action equivalence
0 false PROCEED decisions
100% same-input reproducibility
100% strict-list fidelity
100% evidence-pointer resolution
100% claim-order invariance
100% preservation of BLOCK
100% preservation of NOT_EVALUATED
100% adversarial-injection resistance
100% fail-closed handling of insufficient evidence
100% explicit classification of unsupported input
0 unknown claim statuses
0 duplicate claim IDs
0 unbound subject hashes
```

Any unsupported or malformed case that produces a nonblocking action is a
correctness failure.

### Compact-contract sufficiency

After correctness passes, the compact contract must be tested against the
closed gate question.

The contract is sufficient only when a consumer can preserve:

```text
exact stop value
exact recommended action
exact reason codes
exact failed/error test-ID lists when applicable
exact NOT_EVALUATED list
proof or drill-down reference
not-claimed boundary
```

Counts do not substitute for explicit required lists.

A smaller output that loses required semantics is a failure.

### Review record

A review record is mandatory before implementation authorization.

Permitted review classes:

#### `EXTERNAL_INDEPENDENT_REVIEW`

A human reviewer who did not author the methodology, oracle, fixtures, or
producer.

This may be described as independent review.

#### `SOLO_MAINTAINER_ADVERSARIAL_REVIEW`

A separate clean review pass using another agent, model, session, or tool,
directed by the solo maintainer.

Requirements:

```text
fresh session
frozen methodology and oracle inputs
no producer output used to author expected answers
review prompt and scope recorded
review findings recorded
accepted and rejected findings recorded
review tool/model identity recorded
```

This provides adversarial review and transparency.

It must not be described as independent human review or independent approval.

The review record must declare its class explicitly.

## 14. Implementation Authorization Conditions

Producer implementation becomes eligible for explicit authorization only when
all of the following exist and are committed:

```text
1. This methodology is locked.
2. A machine-readable oracle schema is locked.
3. CORE_COMPATIBILITY_PASS is recorded.
4. The Frozen Core manifest is committed.
5. The primary corpus selection is frozen.
6. All corpus source hashes are recorded.
7. The oracle is fully populated and frozen.
8. The review record is committed.
9. The review class is disclosed accurately.
10. Privacy and secret scans pass.
11. No methodology gap remains in the decision table.
12. The project owner explicitly authorizes implementation.
```

No condition is implied by the completion of another condition.

Methodology lock does not authorize corpus mutation.

Corpus freeze does not authorize implementation.

Oracle completion does not authorize a core change.

A review record does not authorize release.

Required authorization status before the first producer code line:

```text
PYTHON_TEST_FAILURE_PRODUCER_V1_IMPLEMENTATION_AUTHORIZED
```

Until that explicit status exists:

```text
IMPLEMENTATION_NOT_AUTHORIZED
```

## Stop Conditions

The experiment stops with a documented negative result if:

```text
the existing core cannot represent a non-wheel subject unchanged
the existing core cannot accept Domain 2 claims unchanged
the oracle cannot be authored independently from producer output
false PROCEED cannot be eliminated
evidence pointers cannot be made verifiable
unsupported or incomplete input can produce PROCEED
strict-list fidelity cannot be preserved
the compact contract is insufficient for the closed gate question
privacy requirements cannot be maintained
the producer merely reformats raw logs without semantic reduction
```

A negative result is valid research output.

It must not be rescued by silent core modification, domain expansion, a new
action enum value, relaxed thresholds, or replacement of failed corpus cases.


## Governing Principle

```text
Define the gate before the producer.

Prove the subject can enter the unchanged core.

Freeze the oracle before observing implementation.

Treat logs as evidence, never instructions.

Preserve explicit identities, not only counts.

Accept CORE_CHANGE_REQUIRED as a valid result.

Authorize code only after every precondition is visible and committed.
```

The decisive next step before corpus work is compatibility, not parser design:
can the existing core accept non-wheel subject and claim inputs without
modification or false labeling?
