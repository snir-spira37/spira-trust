## Current Baseline

The accepted baseline is:

```text

  assessment

  public non-executing dry-run library module

private branch:
  execution-authorization evaluator
  consistency-not-truth
  ACTION_NOT_PERFORMED
  EA-TCB-03 carried in every result
```

The non-executing authorization chain is complete. Any move toward runner
behavior is a new decision class.

## No Generic Runner

A future runner must never be:

```text
generic shell
generic subprocess
generic script runner
generic Python module runner
generic filesystem mutator
generic network client
arbitrary command executor
```

Every future action class must be explicitly named, typed, scoped, and reviewed.
If an action class cannot be described without arbitrary command, arbitrary
path, or arbitrary network parameters, it is not eligible for runner work.

## Required Future Models

Before implementation can be considered, later docs-only gates must define:

```text
action-class taxonomy:
  exact allowed action classes and explicit forbidden classes

side-effect budget:
  precise filesystem, process, network, and state changes allowed per action
  class; "unbounded" is invalid

trusted-verifier / runner separation:
  how the runner receives the exact context digest independently checked under
  EA-TCB-03

operator protocol:
  who initiates the runner after human go and how operator identity is bound

rollback/abort protocol:
  what abort path exists before, during, and after action

audit log model:
  how considered, authorized-for-consideration, initiated, completed, failed,
  aborted, and rolled-back states are distinguished

secret boundary:
  what secrets are forbidden to pass through logs, reports, or action payloads

failure semantics:
  what happens on partial success, timeout, interruption, or inconsistent
  post-action evidence

public claim boundary:
  what can be said if runner code exists, without claiming safety,
  certification, or proof of isolation
```

## Human Go Is Still Not Execution

A future human go remains a prerequisite, not an executable trigger.

Forbidden:

```text
human go automatically invokes runner
human go appears as a CLI flag
human go is read from environment variable
human go is inferred from CI success
human go is inferred from all checks passed
human go is generated by the runner
```

The runner, if ever implemented, must require a separate operator initiation
step whose identity and context are recorded.

## Runner / TCB Separation

The runner must not be the trusted verifier.

Forbidden:

```text
runner computes the digest it then trusts
agent supplies both approved context and runner-intended context
runner accepts target-supplied context as authoritative
runner mutates inputs after verification
runner executes a context not bound by human go
```

If the future design cannot keep the trusted verifier independent from runner
input construction, it must stop with:

```text
RUNNER_TCB_SCOPE_REVISION_REQUIRED
```

## Side-Effect Budget Requirements

A future side-effect budget must classify at least:

```text
process effects
filesystem reads
filesystem writes
filesystem deletes
network reads
network writes
service or host state changes
credential access
logging and audit writes
rollback or abort effects
```

The default is:

```text
side_effect_budget = 0
```

Any non-zero budget requires a later authorization naming the exact action
class and allowed effects.

## Required Future Conformance

Any future runner implementation authorization must require negative tests:

```text
1. no human go -> no runner invocation.
2. human go without operator initiation -> no runner invocation.
3. operator initiation without human go -> no runner invocation.
4. human go digest mismatch -> no runner invocation.
5. runner-intended context mismatch -> no runner invocation.
6. unallowlisted action class -> no runner invocation.
7. arbitrary command parameter -> rejected.
8. arbitrary path parameter -> rejected.
9. arbitrary network target -> rejected.
10. rollback/abort unavailable -> no runner invocation.
11. trusted verifier unavailable -> no runner invocation.
12. nonce replay -> no runner invocation.
13. clock missing or revocation unknown -> no runner invocation.
14. partial failure records audit state and does not claim success.
15. all negative cases produce zero side effects.
```

Positive tests, if ever authorized, must be per-action-class and must prove only
the bounded side effects of that class. They must not prove general safety.

## Public Boundary

Before runner exposure, public text must not say:

```text
safe to execute
approved to run
automatic remediation
severance authorized
isolation proven
runner proves safety
```

Allowed pre-runner language remains:

```text
runner scope not authorized
execution remains not implemented
non-executing authorization chain complete
future runner requires separate scope revision
```
